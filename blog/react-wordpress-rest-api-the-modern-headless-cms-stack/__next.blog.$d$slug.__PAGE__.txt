b:T4150,<p>The traditional WordPress theme architecture — PHP templates rendering HTML on the server, jQuery animations in the front-end — served the web well for two decades. But in 2025, the expectations for web experiences have fundamentally changed. Users expect app-like interactivity, instant page transitions, and real-time updates. This is where the headless WordPress + React stack shines.</p>
<p>In this guide, we&#8217;ll build a production-grade headless WordPress application using Next.js 15, the WordPress REST API, and React 19. You&#8217;ll learn authentication, data fetching patterns, ISR caching, and the trade-offs you need to understand before choosing headless.</p>
<h2>What Is Headless WordPress?</h2>
<p>In a headless setup, WordPress serves exclusively as a content management system (CMS) and API backend. The frontend is a completely separate application — typically built with React, Next.js, Nuxt, Astro, or any other framework. WordPress exposes its data through:</p>
<ul>
<li><strong>REST API</strong> — Built into WordPress core since 4.7. Available at <code>/wp-json/wp/v2/</code></li>
<li><strong>WPGraphQL</strong> — A popular plugin that exposes a GraphQL endpoint at <code>/graphql</code></li>
</ul>
<p>The benefits of going headless are significant:</p>
<ul>
<li>Full control over the frontend tech stack and performance</li>
<li>Better SEO through Next.js SSG/ISR</li>
<li>Superior developer experience (TypeScript, hot reload, modern tooling)</li>
<li>Multi-channel publishing (web, mobile, smart TV) from a single CMS</li>
<li>Separation of concerns — content editors stay in the familiar WP admin</li>
</ul>
<h2>Setting Up WordPress as a Headless CMS</h2>
<h3>Configure CORS</h3>
<p>Your Next.js frontend will make requests from a different domain (e.g., <code>mysite.com</code>) to your WordPress API (e.g., <code>api.mysite.com</code>). You need to configure CORS in WordPress to allow this:</p>
<pre><code class="language-php">// In your theme's functions.php or a mu-plugin
add_action('rest_api_init', function() {
    remove_filter('rest_pre_serve_request', 'rest_send_cors_headers');

    add_filter('rest_pre_serve_request', function($served) {
        $allowed_origins = [
            'https://mysite.com',
            'https://www.mysite.com',
            'http://localhost:3000', // Development
        ];

        $origin = $_SERVER['HTTP_ORIGIN'] ?? '';

        if (in_array($origin, $allowed_origins, true)) {
            header('Access-Control-Allow-Origin: ' . $origin);
            header('Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS');
            header('Access-Control-Allow-Credentials: true');
            header('Access-Control-Allow-Headers: Content-Type, X-WP-Nonce, Authorization');
        }

        if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') {
            status_header(200);
            exit();
        }

        return $served;
    });
}, 15);</code></pre>

<h3>WordPress API Client</h3>
<p>Create a typed API client that wraps all WordPress REST API calls. This centralizes your data fetching logic and makes it easy to swap from REST to GraphQL later:</p>
<pre><code class="language-typescript">// src/lib/wordpress.ts
const WP_API_BASE = process.env.WORDPRESS_API_URL || 'https://api.mysite.com/wp-json/wp/v2';

interface WPPost {
  id: number;
  slug: string;
  title: { rendered: string };
  content: { rendered: string };
  excerpt: { rendered: string };
  date: string;
  modified: string;
  featured_media: number;
  _embedded?: {
    'wp:featuredmedia'?: Array<{
      source_url: string;
      alt_text: string;
      media_details: { width: number; height: number };
    }>;
  };
  categories: number[];
  tags: number[];
}

async function wpFetch<T>(path: string, options?: RequestInit): Promise<T> {
  const url = `${WP_API_BASE}${path}`;
  const res = await fetch(url, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...options?.headers,
    },
    next: { revalidate: 3600 },
  });

  if (!res.ok) {
    throw new Error(`WordPress API error: ${res.status} ${res.statusText} for ${url}`);
  }

  return res.json();
}

export async function getPosts(params?: {
  perPage?: number;
  page?: number;
  category?: number;
  search?: string;
}) {
  const searchParams = new URLSearchParams({
    per_page: String(params?.perPage ?? 10),
    page: String(params?.page ?? 1),
    _embed: 'wp:featuredmedia,author',
    ...(params?.category && { categories: String(params.category) }),
    ...(params?.search && { search: params.search }),
  });

  const posts = await wpFetch<WPPost[]>(`/posts?${searchParams}`);
  return posts.map(transformPost);
}

export async function getPostBySlug(slug: string) {
  const posts = await wpFetch<WPPost[]>(
    `/posts?slug=${slug}&_embed=wp:featuredmedia,author`
  );
  return posts[0] ? transformPost(posts[0]) : null;
}

function transformPost(post: WPPost) {
  return {
    id: post.id,
    slug: post.slug,
    title: post.title.rendered,
    content: post.content.rendered,
    excerpt: post.excerpt.rendered.replace(/<[^>]+>/g, ''),
    date: post.date,
    modified: post.modified,
    featuredImage: post._embedded?.['wp:featuredmedia']?.[0] ?? null,
  };
}</code></pre>

<h2>Authentication for Private Content</h2>
<p>If your WordPress site has membership content, user profiles, or private posts, you need authentication. WordPress supports two main approaches for headless setups:</p>
<h3>Application Passwords (WordPress 5.6+)</h3>
<p>Application passwords are 24-character tokens generated per user in the WordPress admin. They&#8217;re ideal for server-to-server API calls (e.g., your Next.js server fetching private data):</p>
<pre><code class="language-typescript">// Server-side only — never expose in client code
const WP_AUTH = Buffer.from(
  `${process.env.WP_USERNAME}:${process.env.WP_APP_PASSWORD}`
).toString('base64');

const privatePost = await fetch(`${WP_API_BASE}/posts/123?status=private`, {
  headers: {
    Authorization: `Basic ${WP_AUTH}`,
  },
  cache: 'no-store', // Private content should never be cached
});</code></pre>
<h3>JWT Authentication</h3>
<p>For user-facing authentication (e.g., members logging into your site), use the JWT Authentication for WP REST API plugin. This gives you a proper stateless auth flow:</p>
<pre><code class="language-typescript">// Login and get JWT token
async function loginUser(username: string, password: string) {
  const res = await fetch(`${WP_API_BASE.replace('/wp/v2', '')}/jwt-auth/v1/token`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password }),
  });

  if (!res.ok) throw new Error('Login failed');
  const { token } = await res.json();

  // Store securely — httpOnly cookie via Next.js Server Action
  return token;
}

// Authenticated request
async function getPrivateContent(token: string) {
  return wpFetch('/posts?status=private', {
    headers: { Authorization: `Bearer ${token}` },
  });
}</code></pre>
<h2>On-Demand Revalidation Webhook</h2>
<p>ISR with a fixed revalidation interval means new content appears within an hour. For a better editorial experience, implement on-demand revalidation: when a post is published in WordPress, it immediately triggers a cache purge on your Next.js frontend.</p>
<h3>WordPress Side — The Webhook</h3>
<pre><code class="language-php">// In functions.php or a mu-plugin
add_action('save_post', function(int $post_id, WP_Post $post) {
    if ($post->post_status !== 'publish') return;
    if (wp_is_post_revision($post_id)) return;

    $frontend_url  = get_option('headless_frontend_url', 'https://mysite.com');
    $secret        = get_option('revalidation_secret');
    $post_type_map = [
        'post'      => '/blog',
        'portfolio' => '/projects',
    ];

    $base_path = $post_type_map[$post->post_type] ?? null;
    if (!$base_path) return;

    $paths = [$base_path, "$base_path/{$post->post_name}"];

    wp_remote_post("$frontend_url/api/revalidate", [
        'headers' => [
            'Content-Type'   => 'application/json',
            'x-revalidate-secret' => $secret,
        ],
        'body'    => wp_json_encode(['paths' => $paths]),
        'timeout' => 10,
    ]);
}, 10, 2);</code></pre>
<h3>Next.js Side — The Revalidation Endpoint</h3>
<pre><code class="language-typescript">// src/app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';
import { NextRequest, NextResponse } from 'next/server';

export async function POST(request: NextRequest) {
  const secret = request.headers.get('x-revalidate-secret');

  if (secret !== process.env.REVALIDATION_SECRET) {
    return NextResponse.json({ error: 'Invalid secret' }, { status: 401 });
  }

  const body = await request.json();
  const paths: string[] = body.paths ?? [];

  for (const path of paths) {
    revalidatePath(path);
  }

  return NextResponse.json({ revalidated: true, paths });
}</code></pre>

import { useInfiniteQuery } from '@tanstack/react-query';

interface Post {
  id: number;
  slug: string;
  title: string;
  excerpt: string;

async function fetchPosts({ pageParam = 1 }: { pageParam: number }) {
  const res = await fetch(`/api/posts?page=${pageParam}&per_page=9`);

<h2>When NOT to Go Headless</h2>
<p>Headless WordPress is powerful but it&#8217;s not always the right choice. Consider carefully:</p>
<ul>
<li><strong>Page Builder sites:</strong> Elementor, Divi, Beaver Builder — these are deeply coupled to WordPress rendering. Going headless loses all of that.</li>
<li><strong>WooCommerce:</strong> Headless WooCommerce is possible but complex. Most e-commerce sites are better served by a traditional or block-theme approach.</li>
<li><strong>Small content sites:</strong> If you have 5 pages and rarely update content, headless adds unnecessary complexity.</li>
<li><strong>Team without React experience:</strong> Headless requires frontend expertise that your team may not have.</li>
</ul>
<p>The sweet spot for headless WordPress is: content-heavy sites, portfolios, blogs, marketing sites, and apps where performance and developer experience are top priorities.</p>

<p>The WordPress REST API + React/Next.js stack gives you the best of both worlds: WordPress&#8217;s unmatched content management experience for editors, and a modern, performant frontend for users. With ISR, on-demand revalidation, and React Query for interactive features, you can build sites that score 100/100 on Lighthouse while still giving content teams the WordPress experience they love.</p>
<p>The architectural patterns in this guide — typed API clients, transform functions, on-demand revalidation, JWT auth — are production-tested across dozens of projects. Start with a simple blog setup, get comfortable with the data flow, then layer in complexity as your requirements grow.</p>
